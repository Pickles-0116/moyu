
# 《掌柜偷闲录》开发路线图

---

## 一、版本规划

将28项需求划分为 **6个版本**，从核心原型验证到内容扩展，渐进式交付。

---

### v0.5 Alpha — 核心循环验证

| 属性 | 内容 |
|------|------|
| **主题** | "先证明好玩"——用最简UI验证核心循环是否成立 |
| **包含需求** | P0-04 四维资源系统、P0-05 精力系统、P0-03 核心循环、P0-08 事件系统(15事件) |
| **预估工作量** | 13人日 |
| **预估周期** | 单人 ~3周 / 双人 ~2周 |
| **版本目标** | 内部可玩原型：能行进→遇事件→做A/B选择→看资源变化→精力耗尽后等待恢复。UI用最简占位样式，验证"15-30秒核心循环"的体验是否成立 |
| **关键里程碑** | M0: Day 2 资源系统可读写 ✅ → M1: Day 5 核心循环跑通 ✅ → M2: Day 9 事件可触发 ✅ → M3: Day 13 Alpha原型可玩 ✅ |
| **退出标准** | 内部体验核心循环≥5分钟不觉得无聊；A/B选择有意义；精力系统能有效控制节奏 |

> **设计意图**：不投入UI和伪装模式，先用纯逻辑验证"这个游戏到底好不好玩"。如果核心循环不成立，后续投入全是沉没成本。

## 二、P0内部开发优先级排序

基于 **RICE评分 + 依赖关系 + 核心体验闭环** 综合排序：

| 开发顺序 | 需求编号 | 需求 | RICE评分 | 排序理由 |
|----------|----------|------|----------|----------|
| ① | P0-04 | 四维资源系统 | 450 | **地基**：所有系统都依赖资源系统，必须最先完成 |
| ② | P0-05 | 精力系统 | 240 | **节奏控制**：与资源系统紧密耦合，决定玩家"何时停下来" |
| ③ | P0-03 | 核心循环 | 180 | **心脏**：行进→事件→选择→资源变化，依赖①②，是游戏好不好玩的关键 |
| ④ | P0-08 | 事件系统 | 160 | **内容填充**：核心循环的"燃料"，15个事件让循环不空洞 |
| ⑤ | P0-01 | 老板键/伪装模式 | 267 | **差异化核心**：技术独立可并行，但体验上必须与v1.0同步交付 |
| ⑥ | P0-10 | 古风UI界面 | 140 | **皮肤**：可从Day 1并行设计，在v1.0集成。占位UI先行，正式UI后替换 |
| ⑦ | P0-07 | 店铺系统基础版 | 149 | **被动收益**：依赖资源系统，为核心循环增加"经营"维度 |
| ⑧ | P0-06 | 离线收益 | 140 | **留存机制**：依赖资源+店铺，确保玩家"回来有收获" |
| ⑨ | P0-02 | 自动保存 | 225 | **数据安全**：核心系统稳定后接入，避免频繁改存储结构 |
| ⑩ | P0-09 | 首次引导 | 160 | **新手体验**：UI定稿后才能做引导，放最后 |
| ⑪ | P0-11 | 页面性能 | 160 | **收尾优化**：所有功能就位后统一优化，避免反复调优 |

> **RICE评分说明**（1-10标度）：
> - Reach=10（所有玩家都会接触）、Impact=功能对核心体验影响度、Confidence=需求明确度、Effort=工作量
> - RICE = (Reach × Impact × Confidence) / Effort
> - 排序综合考虑RICE得分和硬依赖关系，依赖优先于评分

## 四、关键路径分析

### 4.1 依赖关系图

```
P0-04 资源系统 ──┬──→ P0-05 精力系统 ──→ P0-03 核心循环 ──→ P0-08 事件系统
                 │                                              │
                 ├──→ P0-07 店铺系统 ──→ P0-06 离线收益         │
                 │                                              ↓
                 └──→ P0-02 自动保存 ←── (所有系统稳定后接入) ←──┘
                                                                      ↓
P0-10 古风UI ──→ P0-09 首次引导                              P0-11 性能优化(最后)
                                                                      ↑
P0-01 老板键 ──→ (独立，随时可做，v1.0前集成) ────────────────────────┘
```

### 4.2 三条并行开发轨道

| 轨道 | 职责 | 包含需求 | 说明 |
|------|------|----------|------|
| **Track A：游戏逻辑** | 核心玩法实现 | P0-04→P0-05→P0-03→P0-08→P0-07→P0-06 | 串行依赖链，是关键路径 |
| **Track B：表现层** | UI与引导 | P0-10→P0-09 | 可与Track A并行，UI框架先搭 |
| **Track C：特色功能** | 老板键伪装 | P0-01 | 完全独立，随时可开发 |
| **Track D：基础设施** | 存储与性能 | P0-02、P0-11 | 收尾阶段接入 |

### 4.3 临界路径（Critical Path）

```
P0-04(2d) → P0-05(2d) → P0-03(5d) → P0-08(4d) → P0-07(3d) → P0-06(3d) → P0-02(2d) → P0-09(2d) → P0-11(2d)
 = 25人日（纯串行关键路径）
```

- Track B (P0-10 4d) 在P0-04完成后并行，不延长关键路径
- Track C (P0-01 3d) 完全独立并行，不延长关键路径
- **单人开发**：32人日全部串行 ≈ 6.4周（5d/周）
- **双人开发**：关键路径25d + 协调损耗 ≈ 4周；非关键路径(P0-10+P0-01=7d)由第二人并行消化

### 4.4 可并行开发的需求组合

| 并行组合 | 前提 | 效率提升 |
|----------|------|----------|
| P0-04(资源) ∥ P0-10(UI框架) | 无依赖 | 节省2-3d |
| P0-03(循环) ∥ P0-01(老板键) | 无依赖 | 节省3d |

## 五、风险评估

### 5.1 技术风险

| 风险 | 等级 | 影响 | 概率 | 缓解措施 |
|------|------|------|------|----------|
| **localStorage容量不足** | 🟡 中 | 存档数据超5MB导致丢档 | 30% | ① 设计紧凑的数据结构，预估单存档<100KB ② v1.3提供存档导出功能 ③ 监控localStorage使用量，超80%告警 |
| **单HTML文件体积过大** | 🟡 中 | 加载>3s，影响"打开即玩" | 25% | ① 事件/店铺数据用JSON字符串内嵌，避免多文件 ② CSS精简，避免引入框架 ③ 图片用CSS绘制或Base64内联 ④ v1.0性能优化时统一压缩 |
| **浏览器兼容性** | 🟢 低 | 部分API不支持 | 15% | ① 明确支持Chrome/Edge/Firefox最新版 ② localStorage/Intl等API兼容性已很好 ③ 不使用实验性API |
| **老板键伪装不够逼真** | 🔴 高 | 核心卖点失效，用户不敢用 | 35% | ① 伪装Excel用真实表格样式(行号/列标/公式栏) ② 切换<200ms无闪烁 ③ v1.0阶段做5人以上真实场景测试 ④ v1.3增加多形态伪装 |
| **离线收益计算误差** | 🟡 中 | 玩家感觉收益不对，信任下降 | 20% | ① 用时间戳精确计算 ② 离线收益单独存储，可追溯 ③ 日志记录离线收益明细 |
| **自动保存性能影响** | 🟢 低 | 频繁保存导致卡顿 | 15% | ① 采用防抖(debounce)，状态变更后500ms再保存 ② 只保存变更字段，非全量序列化 |

### 5.2 产品风险

| 风险 | 等级 | 影响 | 概率 | 缓解措施 |
|------|------|------|------|----------|
| **核心循环不好玩** | 🔴 高 | 整个项目无意义 | 25% | ① v0.5 Alpha阶段先验证，不好玩立即调整 ② 控制Alpha投入在13人日以内 ③ 参考《叫我大掌柜》已验证的循环模式 |
| **D1留存<35%** | 🔴 高 | 游戏缺乏生命力 | 40% | ① v1.0确保首次反馈<10秒 ② v1.1上线每日登录奖励+30事件 ③ 监控数据，D1<30%立即分析流失节点 |
| **D7留存<12%** | 🟡 中 | 中期流失严重 | 35% | ① v1.1成就系统+碎片收集提供中期目标 ② v1.2 Prestige系统提供二周目动力 ③ 事件池持续扩充避免内容耗尽 |
| **事件内容重复疲劳** | 🟡 中 | 玩到后期觉得无聊 | 30% | ① v1.0 15事件覆盖5种类型(随机/风险/机遇/人物/灾难) ② v1.1扩至30+ ③ v2.0事件链增加叙事深度 ④ 事件加权随机，降低重复概率 |
| **老板键被识破** | 🔴 高 | 用户不敢在办公室玩 | 30% | ① 伪装界面做到像素级仿真 ② 默认伪装Excel(最常见办公场景) ③ 切换无动画无声音 ④ 快捷键用ESC+空格等隐蔽组合 |
| **Prestige重置劝退** | 🟡 中 | 玩家不愿重来 | 25% | ① 重置保留部分成就/碎片 ② 重置后前期加速(2倍/3倍) ③ 明确展示重置收益 ④ 重置可选非强制 |

### 5.3 风险应对优先级

```
最高优先级（必须在v0.5/v1.0解决）：
  ① 核心循环好不好玩 → v0.5 Alpha验证
  ② 老板键伪装够不够真 → v1.0重点测试
  ③ 首次反馈<10秒 → v1.0性能优化

中等优先级（v1.1/v1.2解决）：
  ④ D1/D7留存达标 → v1.1留存系统
  ⑤ 事件内容疲劳 → v1.1事件扩充
  ⑥ Prestige设计平衡 → v1.2深度成长

低优先级（持续监控）：
  ⑦ localStorage容量 → v1.3存档导出
  ⑧ 浏览器兼容 → 持续测试
```

## 六、利益相关者沟通要点

### 6.1 对开发者的沟通（技术要点与架构建议）

**架构建议：**
- 采用 **状态机模式** 管理游戏状态（行进中/事件中/店铺中/伪装中），状态切换清晰可控
- 游戏数据用 **单一State对象** 集中管理，所有修改通过统一接口，便于自动保存和调试
- 事件数据用 **JSON配置驱动**，事件内容与逻辑分离，后续扩充只改数据不改代码
- 老板键用 **CSS display切换 + DOM缓存** 实现，不用JS重建DOM，确保切换<200ms

**技术要点：**
- localStorage序列化：建议 `JSON.stringify(state)`，预估单存档50-100KB，远低于5MB限制
- 离线收益计算：用 `Date.now()` 时间戳差值，注意处理时钟回拨异常
- 性能优化策略：事件绑定用事件委托；DOM操作批量处理；避免频繁layout thrashing
- 老板键实现：监听 `keydown` 事件，ESC键触发伪装，伪装界面预渲染隐藏在DOM中

**关键决策点：**
- 单文件 vs 多文件：建议 **单HTML文件内嵌CSS/JS**，满足"打开即玩"，文件体积控制在500KB以内
- 是否引入构建工具：建议 **不引入**，纯原生开发，降低维护成本

### 6.3 对设计师的沟通（UI/UX要点）

**核心体验原则：**
- **"一秒切换，天衣无缝"**：老板键是核心卖点，伪装界面必须像素级仿真Excel。重点设计：行号列标、公式栏、选中单元格高亮、默认显示一些"看起来很忙"的数据
- **"古风但不繁复"**：UI古风元素用于氛围营造，但信息层级要清晰。资源栏(银两/声望/体力/商途值)必须一眼可见，A/B选择按钮要足够大易点击
- **"15秒一个决策"**：核心循环15-30秒，意味着玩家注意力窗口很短。每个界面最多3个可操作元素，减少认知负荷

**关键UI规格：**

| 界面 | 设计要点 | 优先级 |
|------|----------|--------|
| 主游戏界面 | 顶部资源栏(4维) + 中部行进/事件区 + 底部操作区(行进/店铺/背包) | v1.0 |
| 事件弹窗 | A/B两个选项按钮足够大(≥48px高)，显示选项后果预览(资源变化图标) | v1.0 |
| 伪装Excel | 仿真Excel界面：工具栏+列标行号+网格+单元格内容，默认显示财务数据 | v1.0 |
| 店铺界面 | 3种店铺卡片展示，显示等级/收益/升级费用，一键收菜 | v1.0 |
| 首次引导 | ≤3步浮层引导：①点击行进 ②做A/B选择 ③查看资源变化，可跳过 | v1.0 |
| 商途地图 | 江南→中原→西域路线图，已解锁/未解锁状态区分 | v1.2 |
| 成就面板 | 成就列表+进度条+已完成标记 | v1.1 |

**视觉规范建议：**
- 主色调：暖黄/赭石(古风温暖感) + 深棕(沉稳)
- 辅助色：银两=金色、声望=红色、体力=绿色、商途值=蓝色
- 字体：标题用楷体/宋体，正文用系统默认无衬线(保证可读性)
- 伪装模式：完全切换为Excel配色(白底+灰绿网格+蓝色选中框)
