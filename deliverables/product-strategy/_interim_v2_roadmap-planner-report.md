## TL;DR

- **总体判断**：析客的三版本拆分结构（v2.0经济修复→v2.1玩法扩展→v2.2终局深化）**逻辑正确、方向合理**，但存在3处需调整的版本归属问题和1处工作量估算不一致
- **关键发现**：22项需求的个体估算之和为43人日，而文档声称的版本总量为52人日，差额9人日未说明来源；§6.3优先级统计表与§6.1需求清单存在多处对不上
- **调整建议**：将V2-18(防沉迷)/V2-19(文案)/V2-20(整合)从v2.2前移至v2.1，使v2.1成为"完整玩法+配套"的自洽版本，v2.2聚焦终局
- **最高风险**：v2.1负载过重（9项需求跨5个系统）、赌坊掷骰子EV=0.737远低于0.90目标、v2.0经济修复后至v2.1新消耗渠道上线前的"经济过紧"窗口期

## 1. 版本规划评估

### 1.1 对析客版本规划的总评

| 评估维度 | 结论 | 说明 |
|---------|------|------|
| 版本拆分逻辑 | ✅ 合理 | "修复→扩展→深化"的三段式符合产品演进规律 |
| 优先级分配 | ⚠️ 基本合理，需微调 | P0集中于v2.0正确，但v2.1/v2.2边界需调整 |
| 工作量估算 | ❌ 存在不一致 | 个体估算之和(43人日) ≠ 版本总量(52人日)，差额9人日未说明 |
| 版本主题一致性 | ⚠️ v2.1/v2.2主题模糊 | v2.2混入了本应属于v2.1的整合/文案/防沉迷 |

### 1.2 逐版本评估

#### v2.0 经济修复（~10人日）— ✅ 同意，无需调整

**析客规划**：V2-01(囤货约束) + V2-02(增量产出) + V2-03(地点框架) + V2-04(酒楼) + V2-11(扩容声望) + V2-12(离线调整)

**评估**：
- 逻辑自洽：先修复经济破绽（囤货套利占产出37%），再铺设后续系统的地基
- V2-03(地点框架)作为所有后续地点功能的前置依赖，放在v2.0正确
- V2-04(酒楼)虽是新功能而非"修复"，但它是声望获取渠道——没有酒楼，V2-11(扩容声望门槛)就无意义。两者绑定在v2.0是正确的依赖处理
- V2-12(离线调整)降低挂机产出，与四重约束形成"开源节流"的经济修复组合拳

**个体估算验证**：2+1+2+3+1+1 = 10人日 ✅ 与版本总量一致

#### v2.1 玩法扩展（析客称~22人日）— ⚠️ 同意主体，但需扩入3项配套需求

**析客规划**：V2-05(御街) + V2-06/07/08(赌坊三玩法) + V2-09(黑市) + V2-10(医馆) + V2-15/16/17(黑市令+原石+保底)

**评估**：
- 版本主题正确：新增独立玩法系统是v2.1的核心使命
- **问题1：缺失配套需求**。V2-18(防沉迷)、V2-19(古风文案)、V2-20(事件整合)被推到了v2.2，但它们是赌坊/黑市/地点系统的"配套设施"，不应与系统开发脱节：
  - V2-18(防沉迷)是赌坊的安全阀——赌坊上线时不带防沉迷 = 合规风险
  - V2-19(文案)应在每个系统开发时同步撰写，批量补写v2.0/v2.1文案是低效的返工
  - V2-20(整合)让"黑市商人"事件掉落黑市令、"赌坊诱惑"事件跳转赌坊——这是v2.1新系统与现有系统的衔接，推迟到v2.2意味着v2.1的赌坊/黑市与事件系统割裂了一整个版本
- **问题2：负载偏重**。9项需求跨5个系统（御街/赌坊/黑市/医馆/黑市令），对单人开发是高风险的并行负载

**个体估算验证**：2+3+2+4+4+2+1+1+1 = 20人日（析客称~22，差2人日可视为版本内整合开销）

**调整建议**：将V2-18(1人日)、V2-19(1人日)、V2-20(2人日)前移至v2.1，使v2.1成为"玩法+配套"的完整自洽版本。调整后v2.1 = 20+4 = 24人日

#### v2.2 终局深化（析客称~20人日）— ⚠️ 同意，但移出3项后需重新定位

**析客规划**：V2-13(盐场) + V2-14(供奉) + V2-18(防沉迷) + V2-19(文案) + V2-20(整合) + V2-21(幸运) + V2-22(庆典)

**评估**：
- V2-13(盐场)和V2-14(供奉)作为终局长线目标，放在v2.2正确
- V2-21(幸运)和V2-22(庆典)是声望系统的增强，独立性好，放在v2.2合理
- 移出V2-18/19/20后，v2.2聚焦"终局消耗出口+声望增强"，主题更纯粹

**个体估算验证**：5+2+1+1+2+2+2 = 15人日（析客称~20，差5人日）

调整后v2.2 = V2-13(5) + V2-14(2) + V2-21(2) + V2-22(2) = 11人日

### 2.4 优先级洞察

1. **快赢区（高价值低难度）**是v2.0的核心：V2-01/02/11三项RICE评分最高（64/64/96），应确保在v2.0首批完成
2. **战略投入区（高价值高难度）**的4项（V2-04/08/09/13）是v2的"灵魂功能"，决定v2是否值得做——但V2-13(盐场)RICE仅4.8，投入产出比最低，放在v2.2末位正确
3. **V2-12(离线调整)**用户价值仅2分（对玩家是纯负面改动），但实现极简(1人日)且经济修复必须——属于"不得不做但不讨好"的需求
4. **V2-07(竞猜)**用户价值最低(2分)且EV为正(1.10)，与赌坊"负期望消耗"的设计目标矛盾——建议降低其优先级或限制其影响

## 4. 里程碑与发布标准

### 4.1 v2.0 经济修复 — 里程碑M1

**发布标准**：

**致开发同学，v2路线图概要：**

**1. v2分3个版本，共约55人日（含缓冲），优先级明确**

| 版本 | 主题 | 人日 | 核心交付 | 何时做 |
|------|------|------|---------|--------|

| v2.1 | 玩法扩展 | 29 | 御街+赌坊三玩法+黑市+医馆+配套 | v2.0验收后 |
| v2.2 | 终局深化 | 14 | 盐场+供奉+幸运+庆典 | v2.1验收后 |

**2. 三个"必须先做"的理由**
- v2.0必须先做：经济已破（囤货套利占产出37%），不修复就加新系统=在通胀地基上盖楼
- V2-03(地点框架)必须先做：所有后续地点功能(酒楼/赌坊/黑市/医馆)都依赖这个入口
- V2-01(囤货约束)+V2-02(增量产出)可立即并行：无依赖，参数级修改，2-3天可见效

**3. 三个"不要踩的坑"**
- 赌坊EV还没调好：掷骰子综合EV=0.737(目标0.90)，开发前需先确定豹子赔率调整方案
- v2.1别想一次做完：9-12项需求跨5系统，建议准备拆分预案(赌坊优先，黑市可拆为v2.1.5)
- 文案别攒到最后：V2-19(古风文案)应跟系统同步写，不要等全部功能做完再批量补

**4. 三个"关键验收数字"**
- v2.0验收：囤货日均套利从~107万降到~12万(降幅89%)
- v2.1验收：赌坊赌石EV≈0.905，黑市查抄概率≈5%
- v2.2验收：满级余额率从89%降到≤20%

**5. 版本间不要跳过验收**
- v2.0经济没修好 → v2.1的消耗渠道白搭
- v2.1赌坊EV没达标 → v2.2的盐场平衡数据不可信
- 每个版本都有经济平衡验收项，请务必跑模拟测试再进入下一版本
