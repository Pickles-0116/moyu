## 模块一：挂机收益降低

### 1.1 设计目标

降低挂机/离线收益占比，同时不损害碎片化玩家的核心体验。数析报告指出离线收益占总产出34-49%，是经济模型的重要支柱；竞析报告警示过度削弱离线收益会损害D1/D7留存。

**核心策略**：
1. 离线收益：固定比例降至35%/4h（Prestige后50%/6h）——数析推荐参数
2. 在线被动：引入"操作激活"机制——不直接削弱店铺产出，而是要求最低操作频率维持100%效率
3. 在线独占：新增"商机闪现"随机奖励——用正向激励（而非纯惩罚）鼓励在线

### 1.2 离线收益参数调整

#### 1.2.1 参数对照表

| 参数 | v2当前值 | v2.3值 | 降幅 | 理由（数析依据） |
|------|---------|--------|------|----------------|
| 离线收益比例(无Prestige) | 50% | **35%** | -15pp | 适度削弱，日总产出-11.1% |
| 离线收益上限(无Prestige) | 6h | **4h** | -2h | 减少隔夜挂机产出 |
| 离线收益比例(Prestige后) | 60% | **50%** | -10pp | 保持Prestige激励差(50% vs 35%) |
| 离线收益上限(Prestige后) | 8h | **6h** | -2h | 同步调整 |

#### 1.2.2 方案选择：固定比例 vs 分段递减

**决策：采用固定比例方案（数析推荐），不采用分段递减（竞析建议）**

| 对比维度 | 固定比例(35%/4h) | 分段递减(0-2h:35%→2-4h:20%) |
|---------|-----------------|---------------------------|
| 实现复杂度 | 🟢 低（改2个参数） | 🟡 中（需分段计时逻辑） |
| 玩家理解 | 🟢 直观（"离线拿35%") | 🟡 不直观（"前2小时35%后2小时20%"） |
| 等效效率 | 35%（4h全段） | 27.5%（加权平均） |
| 日总产出影响 | -11.1%（数析验证可接受） | -15%+（过度削弱） |
| 竞品先例 | Cookie Clicker比率折扣 | ⚠️ 竞析报告指出"主流Idle游戏中罕见" |

**选择理由**：
1. 固定比例35%/4h恰好处于数析分析的"回来领一下有惊喜感但不构成主要产出"区间
2. 分段递减等效效率27.5%低于数析推荐的35%基准，可能导致D1留存下降
3. 竞析报告本身指出分段递减"在主流Idle游戏中罕见"，缺乏直接先例验证
4. 摸鱼用户画像（日均6-8次×2-5min）每次操作即可领取离线收益，4h上限已足够覆盖
5. Prestige前后差异（35% vs 50%、4h vs 6h）保持激励梯度

#### 1.2.3 离线收益计算逻辑修改

```javascript
// 修改 calculateOfflineEarnings() 方法
calculateOfflineEarnings() {
  const now = Date.now();
  const lastActive = this.state.lastActiveTimestamp || now;
  const offlineSeconds = Math.floor((now - lastActive) / 1000);

  // v2.3: 根据Prestige状态选择参数
  const isPrestiged = this.state.prestige.count > 0;
  const maxHours = isPrestiged
    ? GAME_CONFIG.offlineMaxHoursPrestiged  // 6h
    : GAME_CONFIG.offlineMaxHoursBase;       // 4h
  const ratio = isPrestiged
    ? GAME_CONFIG.offlineRatioPrestiged      // 0.50
    : GAME_CONFIG.offlineRatioBase;           // 0.35

  const cappedSeconds = Math.min(offlineSeconds, maxHours * 3600);
  const effectiveHours = cappedSeconds / 3600;

  // 计算各店铺离线产出
  let totalEarnings = 0;
  for (const shopId in this.state.shops) {
    const shop = this.state.shops[shopId];
    if (!shop || !shop.unlocked) continue;
    const hourlyOutput = this.getShopOutput(shopId) * 360; // 10s结算→小时产出
    totalEarnings += Math.floor(hourlyOutput * effectiveHours * ratio);
  }

  // 离线精力恢复（不受离线收益比例影响）
  const energyRecovered = Math.min(
    Math.floor(cappedSeconds / GAME_CONFIG.energyRecoveryInterval),
    GAME_CONFIG.maxEnergy - this.state.resources.energy
  );

  return {
    silver: totalEarnings,
    energy: energyRecovered,
    offlineHours: parseFloat(effectiveHours.toFixed(1)),
    ratio: ratio,
  };
}
```

### 1.3 在线被动"操作激活"机制

#### 1.3.1 机制设计

借鉴竞析报告引用的Anti-Idle: The Game设计哲学——"不惩罚挂机者，但奖励主动玩家"。不直接降低店铺产出效率，而是要求玩家有"最低操作频率"才能维持100%产出。

| 机制 | 设计 | 效果 |
|------|------|------|
| 活跃加成窗口 | 每次行进/外出/交易操作激活15分钟"全效窗口" | 窗口内店铺100%产出 |
| 非活跃衰减 | 超过15分钟无操作，店铺产出降至70% | 被动挂机产出-30% |
| 深度衰减 | 超过30分钟无操作，降至50% | 长时挂机产出-50% |
| 伪装模式 | 伪装模式下视为"非活跃"，产出70% | 伪装不能替代操作 |

**操作激活源**（以下操作均刷新15min窗口）：
- 商途行进（点击"出发"）
- 外出地点（前往酒楼/御街/赌坊/黑市/医馆）
- 商情交易（买入/卖出货物）
- 道具使用（快捷栏使用道具）
- 店铺升级

**不激活操作**（以下不刷新窗口）：
- 查看面板（背包/成就/统计）
- 切换伪装模式
- 查看日志

#### 1.3.2 活跃状态计算逻辑

```javascript
// state.activity = {
//   lastActiveTimestamp: 0,       // 上次操作时间戳
//   activeWindowDuration: 900,    // 15分钟=900秒
//   deepInactiveThreshold: 1800,  // 30分钟=1800秒

// 修改 getShopOutput() 方法，加入活跃度系数
getShopOutput(shopId) {
  const shop = this.state.shops[shopId];
  if (!shop || !shop.unlocked) return 0;
  const shopDef = SHOPS[shopId];

  // ... 原有里程碑+增量产出逻辑 ...
  let output = milestoneOutput + incrementalOutput;
  output = Math.floor(output * this.state.journey.shopOutputBoost);

  // Prestige legacy加成
  if (this.state.prestige.legacy > 0) {
    output = Math.floor(output * (1 + this.state.prestige.legacy * GAME_CONFIG.legacyPerReputation));
  }

  // v2.3新增：在线被动活跃度系数
  if (!this.state.disguise.active) {
    // 非伪装模式：根据活跃度计算系数
    const activityMultiplier = this.getActivityMultiplier();
    output = Math.floor(output * activityMultiplier);
  } else {
    // 伪装模式：固定70%
    output = Math.floor(output * GAME_CONFIG.disguiseOutputMultiplier);
  }

  return Math.floor(output);
}

// 新增方法
getActivityMultiplier() {
  const now = Date.now();
  const lastActive = this.state.activity.lastActiveTimestamp || now;
  const elapsed = (now - lastActive) / 1000; // 秒

  if (elapsed <= GAME_CONFIG.activeWindowDuration) {
    // 15分钟内：全效
    return 1.0;
  } else if (elapsed <= GAME_CONFIG.deepInactiveThreshold) {
    // 15-30分钟：70%
    return GAME_CONFIG.inactiveOutputMultiplier;
  } else {
    // 30分钟以上：50%
    return GAME_CONFIG.deepInactiveOutputMultiplier;
  }
}

// 操作时调用
refreshActivity() {
  this.state.activity.lastActiveTimestamp = Date.now();
}
```

### 1.4 "商机闪现"在线独占事件

#### 1.4.1 机制设计

借鉴竞析报告引用的Cookie Clicker金色饼干机制——离线无法获得的高价值随机奖励，创造在线机会成本。

| 参数 | 值 | 说明 |
|------|------|------|
| 触发条件 | 在线且处于活跃窗口内（15min内有过操作） | 伪装模式/非活跃不触发 |
| 触发概率 | 5%/次结算（每10秒店铺结算时判定） | 约每3.3分钟触发1次 |
| 效果持续时间 | 30秒 | 短暂高收益窗口 |
| 效果 | 店铺产出×2 | 在线独占增益 |
| 视觉反馈 | 主界面闪现金色光效+"商机闪现！"飘字 | 即时感知 |
| 与现有事件系统关系 | 独立于EVENT_POOL，不影响商途事件触发 | 纯增益叠加层 |

#### 1.4.2 实现代码

```javascript
// 新增 state 字段

//   ...,
//   businessFlashActive: false,
//   businessFlashEndTime: 0,

// 在每10秒店铺结算逻辑中新增
settleShops() {
  // v2.3: 商机闪现触发判定
  if (!this.state.disguise.active &&
      this.isWithinActiveWindow() &&
      !this.state.activity.businessFlashActive) {
    if (Math.random() < GAME_CONFIG.businessFlashChance) {
      this.state.activity.businessFlashActive = true;
      this.state.activity.businessFlashEndTime = Date.now() + GAME_CONFIG.businessFlashDuration * 1000;
      this.addLog('✨ 商机闪现！接下来30秒店铺产出翻倍！', 'success');
      this.emit('businessFlash');
    }
  }

  // 检查商机闪现是否过期
  if (this.state.activity.businessFlashActive && Date.now() > this.state.activity.businessFlashEndTime) {
    this.state.activity.businessFlashActive = false;
  }

  // 店铺结算
  for (const shopId in this.state.shops) {
    const shop = this.state.shops[shopId];
    if (!shop || !shop.unlocked) continue;
    let output = this.getShopOutput(shopId);

    // v2.3: 商机闪现×2加成
    if (this.state.activity.businessFlashActive) {
      output *= GAME_CONFIG.businessFlashMultiplier;
    }

    this.modifyResource('silver', Math.floor(output));
  }
}
```

## 模块二：道具动态定价

### 2.1 设计目标

通过升级店铺降低道具购买费用，给玩家"升级有额外回报"的正反馈。竞析报告引用Realm Grinder的Goblin血统（降低建筑成本倍率）作为直接先例，大商途的"原石价格∝现金"作为动态定价哲学参考。

### 2.2 动态定价公式

采用数析推荐参数：基础费+30%、每10级降4%、Lv.55回平、Lv.100降22%。

#### 2.2.1 核心公式

```javascript
/**
 * 计算道具动态购买价格
 * @param {string} itemId - 道具ID
 * @returns {number} 动态购买价格
 */
function getItemDynamicPrice(itemId) {
  const itemDef = ITEM_DEFINITIONS[itemId];
  if (!itemDef) return 0;

  const basePrice = itemDef.price;

  // 检查道具是否绑定店铺（仅店铺售卖的道具受动态定价影响）
  const shopId = ITEM_SHOP_BINDING[itemId];
  if (!shopId) {
    // 非店铺售卖道具（事件/黑市/Prestige专属）：不受动态定价影响
    return basePrice;
  }

  const shop = state.shops[shopId];
  if (!shop || !shop.unlocked) {
    // 店铺未解锁：原价×1.30（基础加价）
    return Math.floor(basePrice * (1 + GAME_CONFIG.itemPriceBaseMarkup));
  }

  const shopLevel = shop.level;

  // v2.3: 动态定价从Lv.10开始生效（竞析建议）
  if (shopLevel < 10) {
    // Lv.1-9: 基础加价30%，无折扣
    return Math.floor(basePrice * (1 + GAME_CONFIG.itemPriceBaseMarkup));
  }

  // 计算折扣档位：每10级一档
  const discountTier = Math.floor(shopLevel / 10) - 1; // Lv.10→tier1, Lv.20→tier2...
  // 实际折扣：每档4%
  const shopDiscount = Math.min(
    discountTier * GAME_CONFIG.itemPriceDiscountPerLevel,
    GAME_CONFIG.itemPriceMaxDiscount // 上限40%（Lv.100时10档×4%=40%）
  );

  // 动态系数 = (1 + 基础加价) × (1 - 店铺等级折扣)
  const dynamicMultiplier = (1 + GAME_CONFIG.itemPriceBaseMarkup) * (1 - shopDiscount);

  return Math.floor(basePrice * dynamicMultiplier);
}
```

|---------|---------|---------|---------|---------|-------------|-------------|--------|

### 2.4 折扣叠加规则

竞析报告建议折扣为**加法叠加**而非乘法叠加，总折扣上限-30%。

#### 2.4.1 折扣来源

| 折扣来源 | 机制 | 当前是否存在 | v2.3状态 |
|---------|------|------------|---------|
| 店铺等级折扣 | 每10级-4%（Lv.10起），上限-40% | ❌ 新增 | ✅ 核心机制 |
| 声望折扣 | 预留接口（如声望≥500时道具-5%） | ❌ | ⏸ 预留（v2.4+） |
| Prestige折扣 | 预留接口（如Prestige≥3时道具-5%） | ❌ | ⏸ 预留（v2.4+） |

#### 2.4.2 叠加规则

```javascript
/**
 * 计算道具最终购买价格（含所有折扣叠加）
 * 折扣为加法叠加，总折扣上限-30%（从+30%基础加价后的价格计算）
 */
function getItemFinalPrice(itemId) {
  const itemDef = ITEM_DEFINITIONS[itemId];
  const basePrice = itemDef.price;
  const shopId = ITEM_SHOP_BINDING[itemId];

  if (!shopId) return basePrice; // 非店铺道具不受影响

  // 基础加价
  const markedUpPrice = basePrice * (1 + GAME_CONFIG.itemPriceBaseMarkup);

  // 店铺等级折扣
  const shop = state.shops[shopId];
  let shopDiscount = 0;
  if (shop && shop.unlocked && shop.level >= 10) {
    const discountTier = Math.floor(shop.level / 10) - 1;
    shopDiscount = Math.min(discountTier * GAME_CONFIG.itemPriceDiscountPerLevel, GAME_CONFIG.itemPriceMaxDiscount);
  }

  // 预留：声望折扣
  let repDiscount = 0;
  // if (state.resources.reputation >= 500) repDiscount = 0.05;

  // 预留：Prestige折扣
  let prestigeDiscount = 0;
  // if (state.prestige.count >= 3) prestigeDiscount = 0.05;

  // 加法叠加，总折扣上限
  const totalDiscount = Math.min(
    shopDiscount + repDiscount + prestigeDiscount,
    GAME_CONFIG.itemPriceTotalDiscountCap // -30%
  );

  const finalPrice = Math.floor(markedUpPrice * (1 - totalDiscount));
  return finalPrice;
}
```

#### 2.4.3 折扣叠加示例

| 场景 | 店铺等级折扣 | 声望折扣 | Prestige折扣 | 总折扣 | 最终系数 | 金创药价(原50) |
|------|------------|---------|-------------|--------|---------|--------------|
| 前期(Lv.5,无折扣) | 0% | 0% | 0% | 0% | 1.30 | 65 |
| 中期(Lv.30) | 12% | 0% | 0% | 12% | 1.144 | 57 |
| 中期(Lv.50) | 20% | 0% | 0% | 20% | 1.04 | 52 |
| 后期(Lv.80) | 32% | 0% | 0% | 30%(cap) | 0.91 | 46 |
| 后期(Lv.100) | 40% | 0% | 0% | 30%(cap) | 0.91 | 46 |
| 未来(Lv.100+声望500) | 40% | 5% | 0% | 30%(cap) | 0.91 | 46 |

**注意**：当店铺等级折扣达到30%（约Lv.75）后，即使继续升级，总折扣也锁定在30%上限。Lv.100时店铺折扣40%但被cap到30%。这意味着道具价格最低为原价的91%（1.30×0.70=0.91），而非78%。**修正**：数析报告中Lv.100降22%的计算未考虑-30%总折扣上限。实际实施时，总折扣上限-30%意味着最终系数最低为1.30×0.70=0.91，即最低降价9%。

**设计决策**：考虑到数析报告中Lv.100降22%的参数更有激励效果，且当前无其他折扣来源，**v2.3暂不启用-30%总折扣上限**，仅保留接口。总折扣上限将在声望/Prestige折扣实装时启用。当前仅店铺等级折扣单独生效，最高40%（Lv.100时系数0.78，降22%）。

### 4.2 变卖折损计算公式

综合数析建议（累计投入×40%）和竞析建议（50%基础+等级递减），采用以下方案：

#### 4.2.1 公式设计

```javascript
/**
 * 计算店铺变卖回收金额
 * @param {string} shopId - 店铺ID
 * @returns {object} { recoveryAmount, recoveryRate, details }
 */
function calculateShopSellRecovery(shopId) {
  const shop = state.shops[shopId];
  if (!shop || !shop.unlocked) return null;

  // 1. 计算累计升级投入
  const cumulativeInvestment = calculateCumulativeUpgradeCost(shopId, shop.level);

  // 2. 计算基础回收率（根据全局变卖次数递减）
  const sellCount = state.shopSelling.totalSellCount || 0;
  let baseRecoveryRate = GAME_CONFIG.shopSellBaseRecoveryRate; // 40%
  baseRecoveryRate -= sellCount * GAME_CONFIG.shopSellRecoveryDecayPerSell; // 每次-10%
  baseRecoveryRate = Math.max(baseRecoveryRate, GAME_CONFIG.shopSellRecoveryFloor); // 最低20%

  // 3. 等级递减折损（Lv.10以上每级-0.2%）
  let levelDecay = 0;
  if (shop.level > GAME_CONFIG.shopSellLevelDecayStartLevel) {
    levelDecay = (shop.level - GAME_CONFIG.shopSellLevelDecayStartLevel)
                 * GAME_CONFIG.shopSellLevelDecayRate;
  }

  // 4. 最终回收率
  let finalRecoveryRate = baseRecoveryRate - levelDecay;
  finalRecoveryRate = Math.max(finalRecoveryRate, GAME_CONFIG.shopSellRecoveryFloorTotal); // 最低10%

  // 5. 回收金额
  const recoveryAmount = Math.floor(cumulativeInvestment * finalRecoveryRate);

  return {
    recoveryAmount,
    recoveryRate: finalRecoveryRate,
    baseRecoveryRate,
    levelDecay,
    cumulativeInvestment,
    sellCount,
  };
}

/**
 * 计算从Lv.1到当前等级的累计升级费用
 */
function calculateCumulativeUpgradeCost(shopId, currentLevel) {
  const shopDef = SHOPS[shopId];
  let total = 0;
  for (let lv = 1; lv < currentLevel; lv++) {
    total += Math.floor(shopDef.upgradeBaseCost * Math.pow(shopDef.upgradeCostMultiplier, lv - 1));
  }
  return total;
}
```

**路径3详细分析（买店→等产出→卖店）**：

以茶馆Lv.50为例：
- 累计升级投入：581,810银两
- 变卖回收(首次,32%)：186,179银两
- 净亏损：581,810 - 186,179 = 395,631银两
- 日产出(Lv.50)：59.8银两/10s = 516,672/天
- 产出覆盖净亏损需要：395,631 / 516,672 ≈ 0.77天

**⚠️ 发现问题**：如果玩家在Lv.50运营超过1天后变卖，产出+回收 > 投入，存在正收益。

**解决方案**（数析建议）：变卖所得分期释放——50%即时到账，50%分5天每日释放20%。

```javascript
// 变卖所得分期释放
function sellShop(shopId) {
  const recovery = calculateShopSellRecovery(shopId);
  const instantAmount = Math.floor(recovery.recoveryAmount * 0.50);
  const pendingAmount = recovery.recoveryAmount - instantAmount;

  // 即时到账50%
  this.modifyResource('silver', instantAmount);

  // 50%分5天释放
  state.shopSelling.pendingReleases.push({
    amount: Math.floor(pendingAmount / 5),
    daysRemaining: 5,
    startDate: this.getDateStr(),
  });

  // 店铺锁定
  state.shops[shopId].unlocked = false;
  state.shops[shopId].level = 0;

  // 冷却计时
  state.shopSelling.cooldownUntil = Date.now() + GAME_CONFIG.shopSellCooldownHours * 3600 * 1000;

  // 全局变卖次数+1
  state.shopSelling.totalSellCount = (state.shopSelling.totalSellCount || 0) + 1;

  this.addLog(`🏪 变卖${SHOPS[shopId].name}，获得${instantAmount}银两（剩余${pendingAmount}银两分5天释放）`, 'warning');
  this.autoSave();
}
```

**路径4详细分析（卖高级店→买低级店）**：

以变卖Lv.50茶馆（回收186,179）转投布庄升级为例：
- 茶馆Lv.50产出：59.8/10s
- 布庄Lv.50产出：149.5/10s
- 产出差：89.7/10s = 7,746,720/天
- 但布庄升级到Lv.50需要1,426,762银两
- 变卖所得186,179仅够布庄升级到约Lv.30
- 布庄Lv.30产出：25×3+25×0.02×29=89.5/10s
- 茶馆Lv.50产出：59.8/10s
- 产出差：29.7/10s = 2,566,080/天
- 这是"正确的玩家决策"（投资更高效的店铺），而非套利

**结论**：路径4实际上是合理的玩家策略选择（降级套现转投），变卖折损已确保"先卖后买"必然亏损，玩家只是在不同效率的店铺间做选择。

### 4.5 变卖冷却机制

**决策：采用24h冷却（数析建议），不采用7天冷却（竞析建议）**

| 对比 | 24h冷却 | 7天冷却 |
|------|--------|--------|
| 防套利效果 | ✅ 足够（卖店→买回需重新全价升级） | ✅ 过度 |
| 玩家体验 | 🟢 灵活（次日可再操作） | 🟡 限制过强 |
| 摸鱼调性 | 🟢 符合（碎片化操作） | ❌ 冲突（等待7天） |

**冷却规则**：
- 变卖任意店铺后，24h内不可再次变卖任何店铺
- 冷却计时器在变卖时启动
- 冷却期间变卖按钮灰色禁用，显示倒计时
- 冷却仅限制变卖操作，不影响其他操作（购买/升级/外出等）

### 4.6 与Prestige系统的关系

| 项目 | 变卖 | Prestige重置 |
|------|------|-------------|
| 触发方式 | 玩家主动选择 | 玩家主动选择 |
| 银两 | 部分回收(40%→20%) | 全部清零 |
| 店铺等级 | 单个店铺→Lv.0(锁定) | 全部店铺→Lv.1 |
| 商途值 | 不影响 | 清零 |
| 声望 | 不影响 | 转化为传承点后清零 |
| 道具 | 不影响 | 消耗型清零 |
| 传承点 | 不获得 | 获得(声望×1) |
| 互相影响 | 变卖不触发Prestige | Prestige重置变卖计数器和待释放银两 |

**关键规则**：
1. **变卖不触发Prestige**：变卖只是回收部分银两，不获得传承点
2. **Prestige不计算变卖收益**：Prestige的传承点仅由声望转化，与变卖无关
3. **Prestige重置变卖状态**：Prestige后变卖计数器归零（首次变卖恢复40%回收率），待释放银两清零
4. **变卖后重新购买**：变卖后店铺回到锁定状态，需重新满足解锁条件（资源门槛），重新购买后从Lv.1开始

## 模块五：惩罚性倒退Game Over

### 5.1 设计目标

用户要求"保证前中后期都有可能挂掉"，但竞析和数析都警告完全清档会摧毁留存。主理人裁定：修改v2 Non-goal"不做体力归零=Game Over"为"不做完全清档式Game Over，但引入'惩罚性倒退'机制"。

**核心设计原则**（综合竞析+数析+主理人裁定）：
1. 不做完全清档——保留长线投资（声望/商途值/道具）
2. 引入"惩罚性倒退"——有痛感但可恢复
3. 前期保证恢复渠道——药铺默认开放+医馆v2.0移入+新手保护期
4. 后期致死风险源为"多系统连锁崩盘"——策略性致死而非运气
5. 可选hardcore模式——给硬核玩家提供真Game Over+额外加成
6. 渐进式惩罚优于突然死亡——借鉴Bills Must Be Paid催收员机制

#### 5.2.2 惩罚内容

```javascript
function triggerPunitiveRegression() {
  // 检查新手保护期
  const playHours = getPlayHours();
  const isNewPlayer = playHours < GAME_CONFIG.newPlayerProtectionHours; // 5小时
  const penaltyMultiplier = isNewPlayer ? GAME_CONFIG.newPlayerProtectionRate : 1.0; // 新手惩罚减半

  // 检查保险机制
  const noDeathDays = getConsecutiveNoDeathDays();
  const hasInsurance = noDeathDays >= GAME_CONFIG.insuranceDaysThreshold; // 连续3天未致死
  const insuranceMultiplier = hasInsurance ? GAME_CONFIG.insurancePenaltyReduction : 1.0; // 惩罚减半

  const totalMultiplier = penaltyMultiplier * insuranceMultiplier;

  // 1. 银两损失：当前银两×50%
  const silverLoss = Math.floor(state.resources.silver * GAME_CONFIG.gameOverSilverLossRate * totalMultiplier);
  modifyResource('silver', -silverLoss);

  // 2. 店铺降级：全部店铺等级-5（最低Lv.1）
  const shopLevelLoss = Math.ceil(GAME_CONFIG.gameOverShopLevelLoss * totalMultiplier);
  for (const shopId in state.shops) {
    const shop = state.shops[shopId];
    if (shop.unlocked && shop.level > 1) {
      shop.level = Math.max(1, shop.level - shopLevelLoss);
    }
  }

  // 3. 体力重置：20点（非0，可继续）
  state.resources.stamina = GAME_CONFIG.gameOverStaminaReset;

  // 4. 精力重置：10点（非满）
  state.resources.energy = GAME_CONFIG.gameOverEnergyReset;

  // 5. 保留项：商途值/声望/道具/传闻/成就（不做任何修改）

  // 6. 记录致死次数
  state.gameOverCount = (state.gameOverCount || 0) + 1;
  state.lastGameOverDate = getDateStr();
  state.consecutiveNoDeathDays = 0; // 重置保险计数

  // 7. 触发剧情事件
  triggerStoryEvent('punitive_regression', {
    silverLoss,
    shopLevelLoss,
    multiplier: totalMultiplier,
    isNewPlayer,
    hasInsurance,
  });

  addLog(`💀 体力耗尽，重病倒下！损失${silverLoss}银两，全店降${shopLevelLoss}级。好在商途未断，来日方长。`, 'danger');
}
```

### 5.5 "负债催收"中期机制

借鉴竞析报告引用的Bills Must Be Paid催收员渐进惩罚机制。

#### 5.5.1 触发条件

当玩家银两降至负数（通过高风险事件/赌坊/盐场等消耗超过持有量）时进入"负债"状态。

**注意**：当前系统银两不可为负。需要新增"允许银两为负"的机制改动，或改为"银两为0且无法支付必需消耗时"触发。为简化实现，v2.3采用后者：

**负债触发条件**：银两≤0 且 有未支付的必需消耗（如医馆费用、道具购买等无法完成）

#### 5.5.2 渐进惩罚

| 负债天数 | 惩罚 | 说明 |
|---------|------|------|
| 第1天 | 声望-5/天，体力上限-5/天 | 轻度惩罚，提醒玩家 |
| 第2天 | 声望-10/天(累计-15)，体力上限-10/天(累计-15) | 加重惩罚 |
| 第3天 | 声望-15/天(累计-30)，体力上限-15/天(累计-30)，触发"店铺拍卖" | 强制变卖1个店铺还债 |

```javascript
// 每日结算时检查负债状态
function checkDebtStatus() {
  if (state.resources.silver <= 0) {
    state.debtDays = (state.debtDays || 0) + 1;

    const repLoss = GAME_CONFIG.debtRepPenaltyPerDay * state.debtDays;
    const staminaMaxLoss = GAME_CONFIG.debtStaminaMaxPenaltyPerDay * state.debtDays;

    modifyResource('reputation', -repLoss);
    state.debtStaminaMaxPenalty = (state.debtStaminaMaxPenalty || 0) + staminaMaxLoss;

    addLog(`📉 负债第${state.debtDays}天！声望-${repLoss}，体力上限-${staminaMaxLoss}。尽快偿还债务！`, 'warning');

    // 第3天触发店铺拍卖
    if (state.debtDays >= GAME_CONFIG.debtDaysToAuction) {
      forceShopAuction();
    }
  } else {
    // 银两恢复为正，清除负债状态
    if (state.debtDays > 0) {
      state.debtDays = 0;
      state.debtStaminaMaxPenalty = 0;
      addLog('✅ 债务已清偿，恢复正常状态。', 'success');
    }
  }
}

// 强制店铺拍卖
function forceShopAuction() {
  // 选择最低等级的已解锁店铺强制变卖
  const shops = Object.entries(state.shops)
    .filter(([id, shop]) => shop.unlocked)
    .sort((a, b) => a[1].level - b[1].level);

  if (shops.length === 0) {
    // 无店铺可卖：触发惩罚性倒退
    triggerPunitiveRegression();
    return;
  }

  const [shopId, shop] = shops[0];
  const recovery = calculateShopSellRecovery(shopId);
  modifyResource('silver', recovery.recoveryAmount);

  state.shops[shopId].unlocked = false;
  state.shops[shopId].level = 0;
  state.debtDays = 0;
  state.debtStaminaMaxPenalty = 0;

  addLog(`🏛️ 债主上门！强制拍卖${SHOPS[shopId].name}还债，获得${recovery.recoveryAmount}银两。`, 'danger');
}
```

### 5.7 后期"多系统连锁崩盘"设计

数析报告建议：后期致死风险源为"多系统连锁崩盘"而非自然体力耗尽。竞析报告引用Cultist Simulator的多维死亡设计作为参考。

#### 5.7.1 连锁崩盘路径

```
盐场查封(损失全部投入)
    ↓
银两断裂(无法升级/购买道具)
    ↓
赌坊试图翻盘(连输→银两更少)
    ↓
无法前往医馆恢复体力(银两不足)
    ↓
体力持续下降(负面事件+高风险事件)
    ↓
体力归零 → 惩罚性倒退
```

#### 5.7.2 多系统叠加风险

| 风险系统 | 单独风险 | 叠加效果 |
|---------|---------|---------|
| 盐场查封 | 损失盐场投入(50K-200K) | 银两大幅减少 |
| 赌坊连输 | 日累计可输200万 | 银两耗尽 |
| 黑市查抄 | 损失货款(50K-10M) | 银两减少+声望补偿不足 |
| 高风险事件 | 体力-20~-30 | 体力骤降 |
| 负债催收 | 声望/体力上限下降 | 恢复能力减弱 |

**设计意图**：
- 后期玩家拥有多系统（盐场/赌坊/黑市/高风险事件），每个系统都有风险
- 单个系统的风险不足以致死（有恢复渠道）
- 但多个系统同时出问题时，恢复渠道被切断，导致连锁崩盘
- 这是"资源管理失败的连锁反应"——有策略深度，不是运气致死

#### 5.7.3 连锁崩盘保护机制

| 保护机制 | 设计 | 理由 |
|---------|------|------|
| 化险符 | 体力归零时自动消耗，保留1点 | 给二次机会 |
| 保险机制 | 连续3天未致死→下次惩罚减半 | 奖励谨慎玩家 |
| 盐场主动收手 | 随时可收手退出，回收50% | 避免被动全损 |
| 赌坊防沉迷 | 连输50万冷却2h | 防止赌性上头 |
| 黑市查抄声望补偿 | 查抄时声望+10 | 减轻挫败感 |

### 5.8 修改v2 Non-goal的措辞

**原Non-goal**：
> ❌ 不做体力归零=Game Over（约束存在但不太严厉）

**修改后Non-goal**：
> ❌ 不做完全清档式Game Over——体力归零不会删除存档或清零所有进度。但引入"惩罚性倒退"机制：体力归零将损失50%银两+全店降5级，保留商途值/声望/道具/传闻/成就。另外提供可选hardcore模式（体力归零=强制Prestige重置，声望获取+20%加成）。

### 5.9 与Prestige系统的协调

| 对比维度 | 惩罚性倒退 | Prestige重置 |
|---------|-----------|-------------|
| 触发方式 | 被动（体力归零） | 主动（玩家选择） |
| 银两 | -50% | -100% |
| 店铺等级 | -5级(保留大部分) | 清零(全部重置) |
| 商途值 | 保留 | 清零 |
| 声望 | 保留 | 转化为传承点后清零 |
| 道具 | 保留 | 消耗型清零 |
| 传承点 | 不获得 | 获得(声望×1) |
| 离线收益 | 不影响 | 恢复至Prestige后参数 |
| 心理感受 | "痛但可恢复" | "主动重置换永久加成" |

**协调原则**：
1. 惩罚性倒退的惩罚 < Prestige重置的惩罚（否则玩家会主动Prestige来"避免"被动惩罚）
2. 惩罚性倒退不获得传承点（没有补偿，纯惩罚）
3. 惩罚性倒退保留商途值/声望（保留长线进度，区别于Prestige的"从头再来"）
4. Hardcore模式的Game Over = 强制Prestige（最严厉惩罚，但有+20%声望加成补偿）

**惩罚力度对比**（以中期Lv.50全部店铺为例）：

| 惩罚项 | 惩罚性倒退 | Prestige重置 | 比例 |
|--------|-----------|-------------|------|
| 银两损失 | 50% | 100% | 50% |
| 店铺等级损失 | 5级(约5%总投入) | 全部(100%) | 5% |
| 商途值损失 | 0% | 100% | 0% |
| 声望损失 | 0% | 100%(转化为传承点) | 0% |
| **综合惩罚指数** | **~20%** | **100%** | **20%** |

惩罚性倒退的综合惩罚约为Prestige的20%，远低于竞析建议的"70-80%"。这是有意为之——摸鱼用户对惩罚的容忍度远低于Roguelike用户，过重的惩罚会直接导致流失。

## Non-goals更新

### 修改的Non-goal

**原**（v1 Non-goal）：
> ❌ 不做体力归零=Game Over（约束存在但不太严厉）

**修改为**：
> ❌ 不做完全清档式Game Over——体力归零不会删除存档或清零所有进度。但引入"惩罚性倒退"机制：体力归零将损失50%银两+全店降5级，保留商途值/声望/道具/传闻/成就。另外提供可选hardcore模式（体力归零=强制Prestige重置，声望获取+20%加成）。

#### 惩罚性倒退相关
- ❌ **不做惩罚性倒退的连锁叠加**（连续多次体力归零不叠加惩罚，每次独立计算）
- ❌ **不做惩罚性倒退的永久debuff**（惩罚是即时的一次性损失，不产生持续负面状态，负债催收除外）

#### Hardcore模式相关
- ❌ **不做Hardcore模式的中途降级**（开启Hardcore后当轮Prestige内不可切回普通模式，Prestige重置后可重新选择）

## 开放问题

1. **离线收益是否采用分段递减**：当前采用固定比例35%/4h（数析推荐），竞析建议分段递减（0-2h:35%→2-4h:20%）。建议先用固定比例上线，观察玩家离线行为后再评估是否切换为分段递减。

2. **道具动态定价总折扣上限是否启用**：当前预留-30%总折扣上限接口但未启用（仅有店铺等级折扣时不需要cap）。当声望折扣/Prestige折扣实装时需要启用。建议v2.4+再决定。

3. **高风险事件与路段的关联**：当前高风险事件按游戏阶段(商途值)触发，是否需要与路段(江南/中原/西域)关联？如"沙漠商队"仅在西域路段出现。建议v2.4+实现路段关联。

4. **店铺变卖分期释放的必要性**：数析建议变卖所得50%分期5天释放以防银两冲击。但摸鱼游戏的银两体量远小于大型Idle游戏，是否真的需要分期？建议先实现即时全额到账，观察是否出现经济问题再决定。

5. **负债催收系统的银两为负机制**：当前设计为"银两≤0且无法支付必需消耗时"触发负债，但实现上需要明确哪些消耗是"必需"的。建议v2.3先实现惩罚性倒退和hardcore模式，负债催收作为v2.4内容。

6. **Hardcore模式的声望加成是否需要调整**：当前+20%声望加成是否足够激励玩家选择hardcore？建议上线后观察hardcore模式选择率，低于5%则考虑提高加成。

7. **新手保护期的判定标准**：当前以"总游戏时间<5小时"判定新手保护，是否改为"首次Prestige前"或"商途值<500前"？建议先用游戏时间，简单可控。

8. **多系统连锁崩盘的显式提示**：是否需要在后期玩家面临连锁崩盘风险时给出UI提示（如"您的盐场风险较高，建议收手"）？建议不加显式提示，保持风险的不确定性——这是策略深度的来源。

| 版本 | 主题 | 关键交付 | 新增人日 |
|------|------|----------|---------|

**分版本理由**（数析建议）：
- 5项新需求同时上线可能产生叠加效应（降低挂机+高风险事件+Game Over=前期生存压力骤增）
- v2.3a先上经济调整类（挂机+定价+变卖），给玩家适应期
- v2.3b再上风险机制类（高风险事件+Game Over），此时玩家已适应经济变化
